# ADR-0010 — Triage-inbox state-file schema & dedupe identity

## Context

Spec 011 (`/servo:heartbeat`) is the scheduled front-end of the servo loop. Slice 011-01 (DONE) shipped a strictly read-only `discover` pass that enumerates a target's signals — CI failures (`gh`), open issues (`gh`), recent commits (`git`) — and writes one JSON record per finding to `<target>/.servo/triage/inbox.jsonl`, plus a generated human view `inbox.md`. The records carry `schema_version: 1` and a stable `finding_id` fingerprint, but 011-01 carries **no cross-run state**: re-running `discover` *overwrites* the JSONL (re-emit the current findings), every finding is `status: "open"`, and the `finding_id` is computed but never used to dedupe.

Slice 011-02 (triage-state-spine) turns the inbox into the **state spine**: a stable identity dedupes across runs, a status lifecycle is tracked, `heartbeat.py status` reads it back, and the next heartbeat **resumes** — already-handled findings are not re-surfaced as new work. That makes the inbox an *append-and-update cross-run state file*, the reciprocal of the per-run `state.json` that [ADR-0004](adr-0004-session-state-file-format.md) governs. Its identity scheme (what makes two discoveries "the same finding"), its status lifecycle, and its merge/retention semantics are a **contract later tooling reads** — 011-03 (dispatch) writes outcomes back to it, 011-04 (the whole-heartbeat cost ceiling) sums spend from it, and `heartbeat.py status` and a human reviewer both read it. Changing any of these after data exists is migration-shaped. Decide once, write it down — the same reasoning that produced ADR-0004 for `state.json`.

Spec 011 explicitly deferred four questions to this slice, and they converge here: (1) the **fingerprint scheme** (too coarse merges distinct failures, too fine re-surfaces the same one); (2) **what makes a finding "actionable"** (a deterministic v1 heuristic); (3) **what a commit-derived finding is** (a commit is history, not a request); and (4) **double-fire idempotency** (a Routine that fires twice hits the *shared* inbox concurrently). Guardrail #4 (added at plan review) adds a fifth: discovered issue/commit text is attacker-influenceable, so the record should carry a **provenance marker** that downstream dispatch (011-03) can act on.

This ADR fixes the **whole** inbox schema (including fields 011-03/011-04 populate, reserved now so the contract is stable before data exists), and the dedupe/merge/retention/concurrency semantics. Slice 011-02 implements the discovery-and-read subset; 011-03 implements the dispatch-write subset against the reserved fields.

## Decision

### Location and the machine/view split

Cross-run triage state lives at **`<target>/.servo/triage/inbox.jsonl`** (the machine spine, one JSON record per line) with a generated **`inbox.md`** human view beside it — the same machine-state + generated-view split as `state.json` + `status-board`, reserved in `docs/architecture.md` "Runtime artifacts" alongside `runs/` and `races/`. The view is regenerated from the spine on every write and carries the `<!-- generated by heartbeat.py — do not hand-edit -->` marker. The spine is the source of truth; a human may hand-edit it (it is "reviewable" by design), so the format stays plain greppable/diffable JSONL — never SQLite.

### Schema version

`schema_version` is the **first key** of every record (mirroring `gate.py` / `loop.py` / ADR-0004) and is **bumped 1 → 2** for this slice. Per ADR-0002's "bump on any doubt" bias, the bump is warranted: 011-02 adds required fields, changes the *meaning* of `discovered_at` (now first-seen, preserved), and changes the write discipline from overwrite to merge. `schema_version` is written **per record**, and all records in a file MUST carry the **same** version — a mixed-version file is corruption (a reader refuses with rc=2, `schema_version_mixed`). Per-record (rather than a file header) matches 011-01 and gives line-level robustness.

### Record schema (`schema_version: 2`)

Each line of `inbox.jsonl` is one JSON object. Fields fall into three **volatility classes** that drive the merge (below):

**Immutable** — set once at first sight, never changes for a given `finding_id`:

- `schema_version` (int `2`, first key)
- `finding_id` (str — stable fingerprint; see *Dedupe identity*)
- `source` (`"ci"` | `"issue"` | `"commit"`)
- `provenance` (`"first_party"` | `"contributor"`) — a *fact* about who produced the finding's text; see *Provenance*
- `discovered_at` (ISO-8601 str — **first**-seen time)

**Sticky** — preserved across merge; mutated only by lifecycle events (dispatch, or a human):

- `status` (`"open"` | `"tried"` | `"passed"` | `"skipped"`) — the dispatch lifecycle; the resume axis
- `attempts` (int, default `0`) — dispatch count; incremented by 011-03
- `outcome` (object | `null`, default `null`) — last dispatch result; written by 011-03 (reserved shape below)

**Volatile** — recomputed/refreshed every pass the finding is re-observed:

- `title`, `detail` (str), `evidence` (object) — refreshed to the current observation
- `last_seen_at` (ISO-8601 str) — most recent pass that re-observed this finding
- `actionable` (bool) — the deterministic triage verdict; consulted by 011-03 dispatch
- `actionable_reason` (str) — machine code, e.g. `ci_default_branch` / `ci_non_default_branch` / `ci_default_branch_unknown` / `ci_non_actionable_event` / `issue_open` / `issue_label_<name>` / `commit_context_only`

**Reserved `outcome` shape** (written by 011-03, fixed here so 011-04 is stable): `{ "run_id": <str — the `.servo/runs/<run-id>/` the dispatched loop spent in>, "oracle_status": <str>, "oracle_composite": <float>, "cost_usd": <float>, "dispatched_at": <ISO-8601 str> }`. `outcome.cost_usd` is load-bearing: 011-04's whole-heartbeat **resumable** ceiling (= discovery cost + Σ dispatched-loop costs) can then sum from the **inbox alone**, instead of cross-joining into N `state.json` files. Note `evidence` (refreshed — the *latest* observation of a recurring finding) and `outcome.run_id` (sticky — the specific servo run a past dispatch spent in) are **different namespaces** and a reader must not conflate them; for CI, `evidence.run_url` is a *GitHub Actions* run URL while `outcome.run_id` is a *servo* run id.

### Dedupe identity (the `finding_id` fingerprint)

The 011-01 scheme is **ratified** as the v2 contract: `sha256("<source>|<part>|...")[:16]`, derived only from stable, content-identifying fields — never timestamps, run ordinals, or attempt numbers:

- **CI**: `sha256("ci|" + workflowName + "|" + headBranch)`
- **issue**: `sha256("issue|" + number)`
- **commit**: `sha256("commit|" + sha)`

The `source` prefix prevents cross-source collisions. **Documented coarseness (accepted for v1):** CI identity is workflow + branch, so distinct failure *causes* within one workflow on one branch collapse to a single finding, and a recurring failure spans multiple underlying CI run instances over time (hence `evidence` is "latest observation," not a fixed run). **Job-granularity is deferred:** `gh run list --json` has no `job` field (verified live, gh 2.92.0) — distinguishing two failed jobs in one workflow needs a second `gh run view <id> --json jobs` call and a fingerprint-scheme revision (a future `schema_version` bump). The branch in the CI fingerprint is deliberate: the same workflow failing on `main` vs. a feature branch are genuinely different findings.

### Two orthogonal axes: sticky `status`, recomputed `actionable`

`status` (sticky) tracks *what happened* to a finding; `actionable` (volatile) is *whether dispatch should consider it*, recomputed every pass. They are **separate fields**, not one, because actionability is a function of *current* signal state — removing a `wontfix` label must flip a finding back to actionable — which a sticky field cannot express without losing either the lifecycle or the re-evaluation. The spec's own phrasing ("each **actionable, `open`** finding becomes a candidate") treats them as two predicates. 011-03's dispatch candidate set is therefore `status == "open" AND actionable AND seen-this-pass`.

`actionable` is **stored**, not derived at read-time, because it is a snapshot of the verdict *as of the discovery pass that had the signal data* (the labels present that pass, the default branch resolved that pass); deriving it later risks `status` and dispatch disagreeing if the repo changed between calls. It is meaningful **only** for findings observed this pass; for a finding whose signal has disappeared it is a historical value, not a current claim (the retention rule below means such a finding is `open`-evicted or lifecycle-retained, so this rarely bites a reader).

### Provenance — an immutable trust *fact* (Guardrail #4)

`provenance` records *who produced the finding's text*: `"first_party"` for `source == "ci"` (the project's own pipeline result) and `"contributor"` for `source ∈ {issue, commit}` (free-form text that is attacker-influenceable on any repo accepting outside contributions). It is **immutable** — a pure function of `source`, which is part of the fingerprint, so it cannot change for a stable `finding_id`; a flip would be an invariant violation, which is the strict framing a security-relevant field wants. It is **stored** (a denormalized projection of `source`) so readers don't re-implement the mapping and drift.

Storing the *fact* (provenance), not a *policy verdict* (e.g. a `trusted: bool`), is deliberate and mirrors the `status`/`actionable` split: 011-03 **derives** its treatment from provenance — a `contributor` finding's `title`/`detail`/`evidence` must be framed in the dispatch prompt as untrusted **data, never instructions**. Recording the policy verdict instead would bake one consumer's question into the spine and force a schema bump when a fourth source or an LLM-triage-assisted finding (both flagged in spec 011) needs a different answer. 011-02 only *records* provenance; the injection vector is inert here (no dispatch).

### Merge and retention (one uniform rule for all sources)

On each `discover` (a read-merge-write):

1. Load the existing inbox keyed by `finding_id` (a missing/torn/`<v2` file → treat as empty / rebuild, per *Migration*).
2. Run discovery → fresh findings keyed by `finding_id`.
3. **Merge:** an existing `finding_id` keeps its immutable + sticky fields (`status`, `discovered_at`, `attempts`, `outcome`, `provenance`) and refreshes its volatile fields (`title`/`detail`/`evidence`/`last_seen_at`/`actionable`/`actionable_reason`). A new `finding_id` is appended with `status: "open"`, `discovered_at = last_seen_at = now`, `attempts: 0`, `outcome: null`. **This is resume:** a `tried`/`passed`/`skipped` finding is never reset to `open`.
4. **Retention (the growth bound):** *keep what is live or what earned a lifecycle.* A finding with `status ∈ {tried, passed, skipped}` is **retained** indefinitely (audit trail) even after its signal disappears from discovery; an `open` finding **not re-observed this pass** is **evicted**. This bounds *every* source by one rule.
5. Atomically write `inbox.jsonl` and the regenerated `inbox.md` (the 011-01 `tmp` + `os.replace` discipline).

The retention rule is **uniform** rather than per-source on purpose. The thing that makes a source's inbox grow without bound is a **monotonic, unbounded keyspace**, not the source label: commit SHAs advance every pass, and so does `headBranch` in the CI fingerprint (every feature branch that ever fails CI mints a permanent `finding_id`). A per-source policy ("commits replace, others merge") mis-classifies CI as bounded and tangles two orthogonal concerns (identity-merge vs. growth-bounding). One rule — keep live-or-lifecycle-bearing findings, evict stale `open` noise — bounds all three (the scrolled-out commit is `open` + unseen → evicted, exactly the window-replace effect with no special case) and reads as a single sentence a reviewer can check.

### Status lifecycle

| transition | who | when |
|---|---|---|
| (none) → `open` | discover (011-02) | first sight of a `finding_id` |
| `open` → `open` | discover (011-02) | re-observed; volatile fields refreshed, `last_seen_at` bumped |
| `open` → `tried` | dispatch (011-03) | a loop ran, oracle did **not** pass |
| `open` → `passed` | dispatch (011-03) | a loop ran, oracle passed |
| `open` → `skipped` | human | explicit triage-out (reserved for a future auto-skip; **011-02 never auto-sets `skipped`**) |
| any → `open` | human | manual override (reset a `tried` to retry, re-open a `skipped`) |
| `open` (unseen this pass) → *evicted* | discover (011-02) | retention rule |
| `tried`/`passed`/`skipped` | — (sticky) | preserved across re-discovery; **never auto-re-dispatched** |

A finding gets **one** dispatch attempt in v1: a `tried` finding whose signal *recurs* stays `tried` and is not re-dispatched, even though the problem is live — by design, so an unfixable failure does not burn budget every heartbeat. The sharp edge is named honestly: a persistently failing main-branch build becomes invisible to dispatch after one failed attempt. `last_seen_at` keeps refreshing on `tried`/`passed` findings (they are volatile-when-re-observed), so its freshness is the signal a human — or a future retry-with-backoff policy reading `attempts` — uses to notice recurrence.

### Concurrency — advisory `flock`, scoped honestly

The merge introduces a read-modify-write, so two concurrently-fired `discover` runs can lose an update (both read inbox *n*, both merge, the second `os.replace` wins). 011-02 guards this with `fcntl.flock(LOCK_EX | LOCK_NB)` on a **separate** lock file `<target>/.servo/triage/.inbox.lock` — never on `inbox.jsonl` itself, because `os.replace` swaps the inode and a lock held on the old fd would protect nothing.

- **flock over an `O_EXCL` pidfile**: the kernel releases an `flock` automatically when the holding process dies, so a SIGKILLed `discover` leaves no stale lock — a pidfile would, requiring racy pid-liveness checks.
- **Non-blocking, refuse-don't-wait**: a cron double-fire has nothing unique to add (same signals); refusing avoids two scheduled runs serializing into a slow heartbeat or piling up. On contention, `discover` emits a `lock_contended` stderr breadcrumb and exits **0** — the inbox is being maintained by the other run; nothing failed. Exit 0 keeps 011-01's closed `{0, 2}` contract, and the structured `lock_contended` reason makes the no-op observable without overloading the exit code (the same "remap the code, preserve the cause in a field" move as ADR-0002).
- **Degradation**: `import fcntl` failing (non-POSIX) → skip locking with a one-time breadcrumb; atomic write still prevents torn files, only the lost-update race is unguarded — acceptable under servo's existing bash-`oracle.sh` POSIX assumption. `flock` raising `OSError` → rc=2 (the environment is broken in a way worth surfacing, consistent with the `triage_dir_unwritable` rc=2 paths). On NFS, `flock` may silently no-op → degrades to atomic-write-only, same as the non-POSIX path.
- **Scope honesty**: within 011-02 (discovery only) the lost-update race has little practical bite — both racers compute near-identical merges and 011-02 writes no sticky transitions. The guard is largely *pre-provisioning for 011-03*, where a dispatch-outcome write races a concurrent `discover` merge. It lands here because the RMW (the merge) lands here.

### Migration (two rules; a deliberate divergence from ADR-0004)

- **`discover` (write path):** a record with `schema_version < 2` is **dropped and re-derived** from live signals. This is lossless **only because** a v1 record carries no sticky lifecycle (011-01 wrote `status: "open"` always) — the precondition is stated, not implicit. It is **not** the general mismatch policy: a future bump that would drop real sticky lifecycle (`tried`/`passed`) requires an explicit field-migration or a documented data-loss acknowledgement.
- **`status` (read path, read-only):** must **not** rebuild — it **warns and best-effort displays** a *lower* known version (v1 records are a readable subset) and **refuses (rc=2, `schema_version_unsupported`)** a *higher* unknown version (forward-incompatible).

This diverges from ADR-0004's hard-refuse-on-any-mismatch, and the divergence is the point: `state.json` references an irreplaceable Claude Code session, so refusing a resume is correct; the inbox is fully **re-derivable from live signals**, so discard-and-rebuild on write is safe and friendlier. The migration cost today is ~zero — the inbox is a git-ignored runtime artifact and 011-01 only just shipped, so no v1 data of consequence exists in the wild.

### Actionability heuristic (v1, deterministic, recomputed each pass)

- **CI**: actionable iff `event ∈ {push, schedule}` **AND** `headBranch == <default branch>`. The `event` field is already fetched (`_GH_RUN_FIELDS`); a `pull_request` run is someone's in-progress PR and is never auto-actionable. The default branch is resolved via `gh repo view --json defaultBranchRef` (authoritative — `gh` is already a dependency), falling back to `git symbolic-ref --short refs/remotes/origin/HEAD`, then a `main`/`master` guess, then **unknown ⇒ `actionable: false`** (`ci_default_branch_unknown` — fail toward *not* auto-spending). `gh repo view` is primary precisely because `origin/HEAD` is unset on many real clones; relying on it would silently disable CI dispatch on healthy repos.
- **issue**: actionable unless it carries a non-actionable label (case-insensitive name match against a named, deliberately-conservative, tunable set: `wontfix`, `won't fix`, `duplicate`, `invalid`, `question`, `discussion`). Labels are already in the `gh issue list --json labels` surface.
- **commit**: `actionable: false` always (`commit_context_only`). A recent commit is history/context for the human reviewer, not a dispatch candidate; the actionable-commit signal (a revert, a `FIXME`) is explicitly deferred and the reason code keeps a future detector additive.

Mis-triage is cheap to correct because the inbox is reviewable — first-pass perfection is not the bar (the same posture as 006-01's classifier).

### `heartbeat.py status` subcommand

A new read-only verb (mirroring the `gate.py invoke/audit`, `loop.py`/`hook.py` house pattern). It reads `inbox.jsonl` and prints a human summary by default (counts by status, by source, by actionability, plus the open-actionable candidate list with each candidate's `last_seen_at`) and a machine summary under `--json` (carrying `schema_version`). Closed `{0, 2}` exit contract, like `discover`: `0` = read OK (including an empty/absent inbox), `2` = environment error (target missing, inbox unreadable) or `schema_version_unsupported` (a higher unknown version). It never gates (no exit 1) and never mutates.

## Consequences

**Positive.**

- **Resume works and is cheap.** A stable `finding_id` + sticky `status` means the next heartbeat skips already-handled findings without re-running them; the merge is a dict keyed by 16 hex chars.
- **Growth is actually bounded**, by one rule, across all three sources — unlike ADR-0004's `oracle_score_history`, which is a flagged-but-tolerated unbounded list. The inbox is cross-run, cross-project-lifetime state, so bounding it matters operationally.
- **011-04 is implementable against the inbox alone.** Reserving `outcome.cost_usd` now means the whole-heartbeat resumable ceiling sums dispatched-loop spend from the JSONL, not from a cross-join into `runs/`.
- **The injection surface is named and carried, not assumed away.** `provenance` gives 011-03 a concrete, immutable handle to treat contributor text as data — Guardrail #4 made actionable.
- **Greppable, diffable, hand-editable.** Plain JSONL: `jq` works, `git diff` is meaningful, a human can reclassify a finding by editing one line. None of this holds for SQLite.
- **Double-fire can't corrupt the inbox**, and a crashed run leaves no stale lock.

**Negative.**

- **CI dedupe is coarse** (workflow + branch). Two distinct failures in one workflow+branch look like one finding until a human opens the run. Job-granularity is a known, deferred follow-up gated on a second `gh` call and a schema bump.
- **One dispatch attempt per finding in v1.** A persistently failing build goes invisible to dispatch after one `tried`, until a human resets it. Retry-with-backoff is a named future extension (`attempts` is reserved for it).
- **`flock` correctness assumes a local POSIX filesystem.** On NFS or a non-POSIX host the lock degrades to atomic-write-only (torn-write-safe, lost-update-unsafe). servo already assumes POSIX (the oracle is bash), so the blast radius is small, but it is real.
- **A `schema_version` bump beyond 2 that drops sticky lifecycle needs a real migration** — the v1→v2 discard shortcut does not generalize, and the ADR says so to stop someone reusing it destructively.

**Neutral.**

- **`actionable`/`provenance` are recomputed/derived but stored.** Storing a derived value risks drift if the deriver changes; the immutable/volatile classes and the "all-identical `schema_version`" invariant are the discipline that keeps it honest.
- **`finding_id` is 16 hex chars (64 bits).** Collision-resistant for any plausible per-project finding count; not a cryptographic identifier. Same posture as ADR-0004's run-id entropy discussion.
- **No field-level schema validation on read.** A reader accesses fields directly and the `schema_version` check is the first line of defense (as in ADR-0004). Field validation could land under a future bump.

## Alternatives considered

- **Fold actionability into the four-value `status` (no separate `actionable` field).** Rejected: a sticky `status` can't un-stick when a `wontfix` label is removed, so you lose either the lifecycle or the re-evaluation. The two-axis split is the only model that keeps both, and the spec's "actionable, open" phrasing already assumes two predicates.
- **Per-source merge policy (CI/issue sticky-merge, commit window-replace).** Rejected: it mis-classifies CI as a bounded keyspace when `headBranch` makes it as unbounded as commit SHAs, and it couples identity-merge with growth-bounding. The uniform merge + retention rule bounds all three sources and special-cases none.
- **`trust: bool` (or `trusted`/`untrusted`) policy verdict instead of `provenance`.** Rejected: it bakes one consumer's question (011-03's prompt framing) into the spine and forces a schema bump when a new source or an LLM-triage finding needs a different treatment. Record the fact, derive the policy — the same move as `status`/`actionable`.
- **`O_EXCL` pidfile lock.** Rejected: leaves a stale lock after SIGKILL, requiring pid-liveness checks that themselves race. `flock`'s kernel-released semantics are strictly better for crash safety.
- **Block-and-wait on lock contention.** Rejected: converts an idempotency guard into a queue, risking two slow scheduled passes piling up. A double-fire's second run has nothing unique to contribute — refuse it.
- **Hard-refuse on any `schema_version` mismatch (ADR-0004's policy).** Rejected for the inbox: unlike `state.json`, the inbox is fully re-derivable from live signals, so a friendlier discard-on-write (refuse/warn on read) loses nothing and avoids a dead-end for a git-ignored artifact.
- **Delete disappeared findings outright / keep everything forever.** Both rejected in favor of the scoped retention rule: deleting loses the `passed`/`tried` audit trail; keeping everything grows unbounded. "Keep live-or-lifecycle-bearing, evict stale `open`" is the middle that bounds growth without losing what mattered.
- **SQLite (or another embedded DB) for the spine.** Rejected, same reasoning as ADR-0004: the access pattern is "rewrite one small object per pass," there is no query workload, and JSON stays greppable/diffable/hand-editable while SQLite needs a CLI, its own migration story, and durability tuning.

## Verification

- **Slice 011-02 ACs** map each clause to a test class under `skills/heartbeat/test_heartbeat.py` (the 011-01 mock-`gh`/`git` PATH-injection harness, no live network): record schema v2 + first-key + all-identical-version; uniform merge/dedupe; resume (sticky `status` survives re-discovery); retention (open-unseen evicted, lifecycle-bearing retained); actionability per-source incl. default-branch resolution order and the `event` gate; immutable `provenance`; concurrency (`flock` contention → `lock_contended` + exit 0, `OSError` → rc=2, no-`fcntl` degrade); `status` subcommand (human + `--json`, closed `{0,2}`, warn-lower/refuse-higher); migration (discover discards `<v2` and rebuilds; status warns/refuses); and the 011-01 read-only byte-snapshot + atomic-write + dependency-free regressions.
- **Spec 011 spec-level DoD** dogfoods the full chain (011-05): repeated `discover` passes over a fixture target produce a deduped, resuming inbox whose `status` read-back is stable across runs.
- **Acceptance gate:** this ADR is flipped from Proposed to Accepted at slice 011-02 close-out (DONE), once the schema is implemented and the ACs pass — at which point `docs/architecture.md` "Runtime artifacts" is finalized to the shipped v2 shape.
